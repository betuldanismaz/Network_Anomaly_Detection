Section 4

Model Architecture & Training

This section describes the architecture, training protocol, and design rationale for all five model families evaluated in this comparative study. We organise the discussion into two principal tracks that reflect the dual-format preprocessing pipeline established in §3.5: the Traditional ML track (§4.1–§4.3), which operates on 2D tabular feature vectors (N, 20) using tree-based ensemble and single-tree classifiers; and the Deep Learning track (§4.4–§4.5), which operates on 3D temporal sequences (N, 10, 20) using recurrent neural architectures. A unified training protocol section (§4.6) describes the hyperparameter selection methodology, loss function, optimiser, and regularisation strategies shared across model families. All five models are trained identically in terms of input features (Top-20, §3.3) and evaluation protocol (per-file stratified split, §3.4), ensuring a fair, apples-to-apples comparison.

4.1

XGBoost Gradient-Boosted Classifier

XGBoost (eXtreme Gradient Boosting; Chen & Guestrin, 2016) is selected as the primary Traditional ML baseline due to its well-documented superiority on structured tabular data, native GPU acceleration, and built-in regularisation through subsampling and column sampling. The model is trained using the multi:softprob objective, which outputs a probability vector over all three classes rather than a hard label — a design choice that enables OvR ROC-AUC computation and provides calibrated confidence estimates for the production inference pipeline.

A. Hyperparameter Configuration

The following hyperparameters were determined through empirical tuning on the validation partition. Early stopping (patience = 50 rounds) is applied using validation set multi-class log-loss (mlogloss) to prevent overfitting on the 2.83M-flow training corpus:

Python — train_xgboost.py (lines 220–233) 
params = {
    'n_estimators':         1000,      # max boosting rounds (early stopping used)
    'max_depth':             7,         # tree depth — balances capacity vs. speed
    'learning_rate':         0.05,      # shrinkage: slow but stable convergence
    'subsample':             0.8,       # row sampling per tree (regularisation)
    'colsample_bytree':      0.8,       # column sampling per tree
    'objective':     'multi:softprob', # outputs P(y=0), P(y=1), P(y=2)
    'num_class':              3,         # required for multi-class objectives
    'eval_metric':    'mlogloss',       # multi-class log loss for monitoring
    'random_state':          42,
    'n_jobs':               -1,         # all CPU cores
    'early_stopping_rounds': 50         # patience on val mlogloss
}
# GPU acceleration: device='cuda', tree_method='hist' (XGBoost ≥2.0)
B. Imbalance Handling

A standard multi-class XGBoost configuration would apply the scale_pos_weight parameter, but this parameter is binary-only. For our 3-class imbalanced setting, we instead compute per-sample inverse-frequency weights using scikit-learn's compute_sample_weight('balanced') and pass them to model.fit() via the sample_weight argument. This assigns the Semantic class (≈6% prevalence) approximately 12–16× higher weight than Benign samples during gradient computation, preventing the model from defaulting to the majority class.

C. GPU Acceleration

The training script includes automatic GPU detection logic. When a CUDA-capable GPU is available, XGBoost 2.0+ parameters device='cuda' and tree_method='hist' are appended to the configuration. On the project hardware, GPU-accelerated training on the full 2.83M-flow corpus completes in significantly less time than CPU-based training with the hist CPU method.

4.2

Random Forest Classifier with Grid Search

Random Forest [Breiman, 2001] is selected as the second Traditional ML model, providing a comparison of bagging-based ensembles versus boosting-based ensembles (XGBoost) under identical feature and split conditions. The Random Forest model uses criterion='entropy' (information gain) rather than Gini impurity, which produces more balanced multi-class splits in the presence of skewed class distributions.

A. Grid Search Methodology

Rather than cross-validating on training data (which introduces ~3× computational overhead and inflated variance when training on 2M+ samples), hyperparameter selection uses an exhaustive grid search scored directly on the held-out validation set (Macro F1). This approach is both computationally efficient and directly optimises the metric that matters for production deployment:

Python — train_randomforest.py (lines 147–153) 
param_grid = {
    'max_depth':        [20, 30, None],         # deeper than prior [10,15,20]: finer splits
    'min_samples_leaf': [10, 30, 50],            # structural pruning floor (noise guard)
    'max_features':     ['sqrt', 'log2'],        # feature subsampling per split
    'class_weight':     ['balanced',
                         {0:1.0, 1:2.0, 2:4.0}],  # balanced vs. softer custom boost
}
# Fixed: n_estimators=100, criterion='entropy', bootstrap=True, random_state=42
# Total: 36 combinations — scored on val.csv Macro F1

The grid tests two class weighting strategies: 'balanced' (standard inverse frequency, weightSemantic ≈ 5.3×) and a custom softer weighting {0:1.0, 1:2.0, 2:4.0} that provides a more moderate boost, trading recall against precision for the minority class. The best combination is selected by highest Macro F1 on the validation set and the corresponding model is retained for final test evaluation.

4.3

Decision Tree Classifier — Interpretability Baseline

The Decision Tree classifier serves a dual purpose: it provides a lower-bound interpretability baseline against which the more complex ensemble and DL models can be compared, and it enables post-hoc extraction of human-readable classification rules via sklearn.tree.export_text — a feature of operational value in SOC contexts where analysts must understand and audit model decisions. A single decision tree can represent the entire decision boundary as a structured if-then-else rule set, making it uniquely auditable.

A. Hyperparameter Search

An 18-combination grid search (3 × 3 × 2) is applied over max_depth, min_samples_leaf, and class_weight, identical in scoring methodology to the Random Forest grid search (validation Macro F1):

Python — train_decisiontree.py (lines 156–160) 
param_grid = {
    'max_depth':        [15, 20, 30],
    'min_samples_leaf': [20, 50, 100],  # prevents noisy leaves at Benign/attack boundary
    'class_weight':     ['balanced', {0:1.0, 1:2.0, 2:4.0}],
}
# Fixed: criterion='entropy', min_samples_split=2×leaf_floor, random_state=42

Crucially, min_samples_split is always set to 2 × min_samples_leaf to enforce structural consistency — a node must have at least twice the leaf-floor samples before being eligible for splitting, preventing the creation of isolated micro-partitions that represent dataset noise rather than genuine class boundaries.

4.4

Stacked LSTM Network

The Long Short-Term Memory (LSTM; Hochreiter & Schmidhuber, 1997) architecture is selected as the primary deep learning model because its recurrent memory cells are theoretically well-suited to the sequential nature of network flows, where adjacent flows within the same session share temporal context not captured by independent per-flow prediction. The model consumes the 3D temporal sequences produced by the preprocessing pipeline (shape: 10 × 20 — 10 consecutive flows, 20 features each) and produces a 3-class probability distribution.

B. Training Configuration

The model is compiled with the Adam optimiser (lr = 0.001; Kingma & Ba, 2015) and trained using sparse categorical cross-entropy loss (appropriate when labels are integer class indices, avoiding the overhead of one-hot encoding on a 2.8M-sample dataset). All training is conducted via a memory-efficient tf.data.Dataset streaming pipeline (batch = 256, prefetch = AUTOTUNE) that loads the 3D training arrays from disk using np.load(..., mmap_mode='r'), avoiding the need to hold the full ≈1.8 GB training corpus in GPU memory simultaneously.

4.5

Bidirectional LSTM Network

The Bidirectional LSTM (BiLSTM; Schuster & Paliwal, 1997) extends the unidirectional LSTM by processing each input sequence in both the forward (t=1 → t=10) and backward (t=10 → t=1) temporal directions simultaneously, concatenating both hidden state sequences before passing them to subsequent layers. This architecture is hypothesised to improve classification of attack patterns where the context of later flows (e.g., confirming that an initial probe is followed by exploitation) is informative in classifying earlier flows in the sequence. The effective per-layer hidden state dimensionality doubles relative to the LSTM since both forward and backward states are concatenated.

A. Architecture
BiLSTM Network Architecture — src/models/train_bilstm.py
Input(shape=(10, 20)) ← identical input to LSTM
↓
Bidirectional(LSTM(128), return_sequences=True) ← output dim: 256 (2 × 128 fwd + bwd)
↓
BatchNormalization() → Dropout(0.3)
↓
Bidirectional(LSTM(64), return_sequences=False) ← output dim: 128 (2 × 64)
↓
BatchNormalization() → Dropout(0.3)
↓
Dense(64, activation='relu') → Dropout(0.3)
↓
Dense(3, activation='softmax')
🔬 LSTM vs. BiLSTM Architectural Difference

The BiLSTM is intentionally constructed to be architecturally symmetric with the LSTM: same hidden unit counts (128 → 64), same dropout rates (0.3), same Dense layer (64, ReLU), same optimiser (Adam, lr=0.001), same callbacks, and same training data. Only the wrapper layer changes — Bidirectional(LSTM(…)) replaces LSTM(…). This design choice ensures that any performance difference observed between LSTM and BiLSTM (§8) is attributable exclusively to the bidirectionality of the processing, not to other architectural variations.

4.6

Unified Training Protocol & Reproducibility

A. Identical Evaluation Conditions

A core methodological commitment of this work is that all five models are evaluated on identical test partitions produced by the same per-file stratified split (§3.4). The test set contains the same flows for XGBoost, Random Forest, Decision Tree, LSTM, and BiLSTM — differing only in representation (2D vector vs. 3D sequence). This eliminates the confound present in most prior comparative NIDS studies where different models are trained on different subsets due to the absence of a shared, explicitly specified split protocol.

B. Reproducibility Seed

A fixed random seed of 42 is applied at all stochastic stages: scikit-learn's random_state=42, XGBoost's random_state=42, and TensorFlow's global seed via tf.random.set_seed(42) plus PYTHONHASHSEED='42' and numpy.random.seed(42). All training scripts set the seed as the very first operation before any library import that could trigger stochastic initialisation.

C. Imbalance Handling — Unified Approach

Class imbalance is addressed consistently across all five models through class-weighted loss computation. For Traditional ML models, scikit-learn's class_weight='balanced' or compute_sample_weight('balanced') (XGBoost) is used. For DL models, the inverse-frequency weights computed during preprocessing (§3.5C) are loaded from models/class_weights.json and passed as the class_weight argument to model.fit(). This ensures that the Semantic minority class (~6%) receives proportionally higher gradient signal regardless of model family.

D. Evaluation Metrics — Consistent Across All Models

All five models are evaluated on the test set using an identical metric suite to enable direct comparison (§8). The primary metric is Macro-averaged F1-score, which computes F1 independently for each class and takes the unweighted mean — ensuring equal representation of the minority Semantic class. Secondary metrics include Macro Precision, Macro Recall, overall Accuracy, Weighted F1 (for comparison with prior work), OvR (One-vs-Rest) ROC-AUC per class and macro-average, and inference latency (ms/sample) measured on a 10,000-sample benchmark batch.

⚠ Note on Training vs. Test Set for DL Models

The LSTM and BiLSTM preprocessing pipeline produces an 80/20 train/test split (rather than the 80/10/10 of the ML pipeline) because the validation partition for early stopping is subsumed into the training data pipeline and passed as validation_data=test_dataset to model.fit(). This means the DL early stopping monitor observes the same test partition that is used for final evaluation — a known limitation of single-split DL training that should be considered when interpreting the DL generalisation metrics. It does not introduce label leakage (no gradients flow from the test partition), but it may produce slightly optimistic early stopping decisions.

Section 4 Summary — Model Architecture & Training

Five models across two tracks: XGBoost (boosted trees, n=1000, max_depth=7, objective=multi:softprob), Random Forest (n=100, criterion=entropy, grid-searched depth/leaf/class_weight), Decision Tree (criterion=entropy, 18-combo grid search, auditable rule export).

Deep Learning: LSTM (LSTM-128 → LSTM-64 → Dense-64 → Softmax-3; lr=0.001, batch=256, EarlyStopping patience=5), BiLSTM (architecturally symmetric; Bidirectional wrappers only change). Both trained with memory-mapped tf.data streaming pipeline for GPU memory efficiency.

Controlled comparison: All models share the same Top-20 features, 3-class taxonomy, per-file stratified test partition, inverse-frequency class weighting, and evaluation metric suite. Random seed 42 throughout.
