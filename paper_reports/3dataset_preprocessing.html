Section 3

Dataset, Feature Extraction & Preprocessing

This section describes all data-related decisions made in this work, from raw dataset selection through to the production of the final train/validation/test partitions used by each model family. We describe the CICIDS2017 dataset and its properties (§3.1), the novel 3-class attack taxonomy (§3.2), the CICFlowMeter feature extraction pipeline and feature selection (§3.3), the per-file stratified split methodology that constitutes a key contribution of this work (§3.4), and the two distinct data representation formats produced for traditional ML and deep learning inference (§3.5).

3.1

CICIDS2017 Dataset Overview

The CICIDS2017 dataset, generated by the Canadian Institute for Cybersecurity and described by Sharafaldin et al. [2018], is the benchmark dataset used throughout this work. It was selected over alternatives (KDD99, NSL-KDD, UNSW-NB15) for the convergence of four properties uniquely relevant to our research objectives: (i) a five-day capture schedule that provides temporal diversity across separate source files, enabling our per-file leakage-free splitting methodology; (ii) multi-profile realistic benign traffic generated from 25 abstract user behaviour models derived from real enterprise network analysis; (iii) a rich set of heterogeneous attack categories covering both volumetric and semantic threat classes simultaneously; and (iv) a widely adopted CICFlowMeter-based feature extraction producing 78 bidirectional flow-level features against which our Top-20 selection can be validated.

The Thursday PM file containing only 36 Infiltration flows is the principal motivation for per-file stratified splitting (§3.4): a global random shuffle would routinely exclude all 36 rows from the test set with non-negligible probability.

After label mapping and the removal of rows with unmapped or missing labels (primarily due to CICFlowMeter extraction artefacts described in §3.3), the processed dataset contains 2,830,743 labelled network flows distributed across the three target classes. 3.2

Custom 3-Class Attack Taxonomy

The raw CICIDS2017 labels include 14 distinct attack sub-categories plus the Benign class — a granularity that, while maximally informative in principle, is operationally intractable for several reasons: (i) certain sub-categories contain fewer than 100 flows, rendering reliable per-class model evaluation statistically impossible; (ii) the triage procedure in a real Security Operations Centre does not require per-attack-variant classification, but rather a coarser determination of whether an incident requires a network-level bandwidth response or an endpoint/application-layer investigation; and (iii) fine-grained labels introduce data-sparsity pathologies in preprocessing (particularly for the LSTM sliding window, where extreme minority classes may produce fewer training windows than the window size itself).

We therefore define a 3-class attack taxonomy based on the dominant operational impact mechanism of each attack group, and implement it via a classes_map.json mapping applied uniformly to both the ML and DL preprocessing pipelines:

Table 7. classes_map.json — Raw Label to 3-Class Integer Mapping
Raw CICIDS2017 Label Integer Class Semantic Group Operational Impact Type
BENIGN 0 Benign Normal traffic — no countermeasure required
DDoS 1 Volumetric Bandwidth exhaustion → network-layer mitigation
DoS Hulk · DoS GoldenEye · DoS Slowloris · DoS Slowhttptest 1 Volumetric Resource exhaustion (HTTP/TCP) → network-layer mitigation
Bot 1 Volumetric C&C beaconing → rate-limit / IP block
PortScan 2 Semantic Reconnaissance → firewall rule update
FTP-Patator · SSH-Patator 2 Semantic Credential brute force → account lockout / MFA enforcement
Web Attack — XSS · SQL Injection · Brute Force (web) 2 Semantic Application exploit → WAF rule / patch management
Infiltration 2 Semantic Exfiltration / backdoor → endpoint investigation
Heartbleed 2 Semantic Memory disclosure → patch management
Label normalisation via NFKC Unicode normalisation and whitespace collapsing is applied before mapping to handle CICIDS2017's inconsistent label formatting (e.g. "Web Attack Brute Force" with double space).
🎯 Taxonomy Design Rationale

The Volumetric class groups all attacks whose primary impact is service availability reduction through traffic manipulation — regardless of whether the mechanism is pure flooding (DDoS/DoS) or covert channel abuse (Botnet). The Semantic class groups all attacks whose primary intent is information discovery or system compromise through targeted, intelligent interaction with specific services or credentials. This two-way split of the attack space directly corresponds to the two primary alert routing categories in enterprise SOC playbooks, and is empirically validated by the distinct flow-level feature signatures of the two groups (Table 8).

3.3

Feature Engineering & Extraction

A. CICFlowMeter Bidirectional Flow Feature Extraction

The CICIDS2017 raw data is provided in pre-extracted form, with all 78 bidirectional flow features generated by CICFlowMeter [Lashkari et al., 2017] applied to the original PCAP files. CICFlowMeter extracts statistical descriptors from TCP/UDP flows defined by the 5-tuple (source IP, destination IP, source port, destination port, protocol). Each flow record captures duration, packet count, byte volume, flag counts, inter-arrival times, and subflow statistics in both the forward (client→server) and backward (server→client) directions. The complete 78-feature space includes flow duration, packet length statistics (min, max, mean, std, variance), inter-arrival time statistics (IAT min/max/mean/std for both directions), TCP flag indicators (SYN, ACK, FIN, RST counts), active/idle time statistics, and derived rate features (packets/s, bytes/s).

⚠ CICFlowMeter Known Artefacts

The CICIDS2017 feature extracts contain two systematic artefacts that must be addressed before model training: (1) Infinite values in Flow Bytes/s and Flow Packets/s arising from zero-duration flow division; (2) NaN values introduced by feature computations on single-packet flows where standard deviation is undefined. Our preprocessing pipeline replaces all ±inf and NaN values with 0.0 before any further processing — a conservative imputation strategy consistent with treating undefined ratios as zero-signal features.

B. Feature Selection — Top-20 CICFlowMeter Features

Training models on the full 78-feature space introduces unnecessary computational overhead and risks diminishing generalisation through noise features. Feature selection was performed via XGBoost feature importance ranking on the full feature set trained on a preliminary binary classification task, followed by validation of the top-N rankings against Random Forest mean decrease in impurity. The final Top-20 feature set, confirmed consistent across both importance measures, is defined in the shared src/config.py module and applied identically to all preprocessing pipelines (ML tabular and LSTM sequential):

This feature set is dominated by backward-direction packet statistics (F01, F02, F05, F06, F07, F11, F15) — a finding consistent with the intuition that server response patterns (the backward direction) are often more discriminative than client request patterns in distinguishing attack traffic from benign activity. Inter-arrival time statistics (F12, F17, F20) capture the temporal rhythm of traffic generation, which is particularly distinctive for automated attacks such as port scanning and brute-force credential enumeration. The mixture of both directions and both volume and timing features provides a balanced, information-rich description of each network flow within a compact 20-dimensional vector.

3.4

Per-File Stratified Split Methodology (Key Contribution C1)

A. The Problem: Temporal Leakage in Multi-Day Datasets

The CICIDS2017 dataset is distributed as eight independent CSV files, each corresponding to a specific capture period with its own attack profile (Table 5). A standard data preparation workflow concatenates all files into a single DataFrame and then applies a random train/test split. We formally argue that this practice constitutes temporal data leakage with respect to this dataset, for two reasons:

Reason 1 — Cross-day temporal correlation: Flows captured on the same day, within the same attack session, share temporal context (common source IPs, destination subnets, session timing patterns). When flows from the same attack session appear in both the training and test partitions, the model's test performance partially reflects memorisation of this shared context rather than generalisation to unseen traffic.

Reason 2 — Minority class starvation: The Thursday PM file contains only 36 Infiltration flows. When all files are merged into a pool of 2.83 million rows and a random 80/20 split applied, the expected number of Infiltration rows in the test set is approximately 7. However, statistical fluctuation means that in a substantial fraction of random seeds, all 36 rows fall into a single partition — producing a test set with zero Infiltration examples and a training set that has seen no corresponding test distribution.

B. The Solution: Per-File Stratified 3-Way Split

Our proposed methodology eliminates both failure modes by reversing the order of operations: stratified splitting is performed independently on each source file, and the resulting per-file split partitions are aggregated after splitting, never before. The procedure is as follows:

Algorithm: Per-File Stratified Split (Applied to Each Source File f ∈ F)
Load file f → Map labels → {0,1,2} → Replace ±inf / NaN → 0 → Stratified split (80/20, seed=42) → Stratified split temp (50/50) → Append to train/val/test buckets

The two-stage split produces proportions of 80% train / 10% validation / 10% test per source file. Stratification is applied at the first split using the 3-class label as the stratification key, guaranteeing that each class's proportional representation within the source file is preserved in all three output partitions. A fallback to non-stratified splitting is implemented for edge cases where the minimum class count within a file is insufficient for stratification (e.g., a single-instance class).

TRAIN — 80%
Scaler fit • Model training
VAL — 10%
Early stopping
TEST — 10%
Final evaluation

After all files are processed independently, the per-file train partitions are concatenated into the global training set, the per-file validation partitions into the global validation set, and the per-file test partitions into the global test set. No flow from any source file can appear in more than one partition.

C. Leakage-Free Feature Scaling

A frequent but subtle source of data leakage in published NIDS work is the fitting of the feature scaler on the combined train+test dataset, effectively allowing test set statistics to influence the transformation applied to training data. Our pipeline enforces a strict protocol: the MinMaxScaler is fitted exclusively on the aggregated training partition and then applied as a fixed transformation to the validation and test partitions. This is implemented consistently in both preprocessing pipelines:

Python — preprocess_ml_3class.py (lines 343–351) 
# Fit scaler on TRAIN ONLY → transform all three (leakage-free)
print("⚖️  Fitting MinMaxScaler on training data only (leakage-free)...")
scaler = MinMaxScaler()
X_train = scaler.fit_transform(X_train)  # fit + transform on train only
X_val   = scaler.transform(X_val)         # transform only (no fit)
X_test  = scaler.transform(X_test)        # transform only (no fit)

# Save scaler for production inference (live_bridge.py uses this)
joblib.dump(scaler, SCALER_PATH)           # → models/scaler_ml_3class.pkl

The fitted scalers are serialised to models/scaler_ml_3class.pkl (for ML models) and models/scaler_lstm.pkl (for LSTM/BiLSTM models) and are subsequently loaded by the production inference pipeline (§6.2) to ensure that live traffic is scaled using exactly the same statistics observed during training — a requirement for deployment correctness that is often incompletely specified in the literature.

✅ Formal Guarantee

No flow from any source file appears in more than one partition. No test-set statistics influence any scaler or model parameter. All three classes are guaranteed to appear in every split for every source file with sufficient class cardinality. These three invariants collectively ensure that the evaluation metrics reported in §8 are free of the contamination artefacts documented in §2.3 and §2.5 for prior work on CICIDS2017.

3.5

Data Representation: 2D Tabular (ML) vs. 3D Temporal Sequences (DL)

The five model families evaluated in this work require two fundamentally different input representations, both derived from the same per-file stratified split but constructed through distinct preprocessing paths.

A. 2D Tabular Format — Traditional ML Models

For XGBoost, Random Forest, and Decision Tree classifiers, each network flow is represented as an independent, identically distributed (IID) feature vector of dimension 20. The training data takes the shape (Ntrain, 20) where each row corresponds to one flow and each column to one of the Top-20 features. After MinMaxScaler normalisation, all feature values are in the range [0, 1]. The output CSV files data/processed_ml/train.csv, val.csv, and test.csv each contain the 20 feature columns plus a Label column (integer 0, 1, or 2). The downstream training scripts consume these files with: X = df.drop('Label', axis=1); y = df['Label'].

B. 3D Temporal Sequences — LSTM & BiLSTM Models

For LSTM and BiLSTM, network flows are not treated as independent samples but as elements of a temporal sequence, with adjacent flows assumed to share temporal context. The preprocessing pipeline constructs fixed-length sequences using a sliding window of length 10 applied with stride 1. Each sequence has shape (10, 20) — ten consecutive flows, each described by the same 20 features — and its label is taken as the label of the final (most recent) flow in the window.

A critical design decision in this pipeline is the order of operations. The stratified raw-flow split is performed before sliding-window construction, ensuring that no window ever spans the train/test boundary:

Python — preprocess_lstm.py (lines 373–393) 
# Step B: STRATIFIED RAW SPLIT (CRITICAL FOR ZERO LEAKAGE)
# We split the RAW data BEFORE creating sequences.
X_train_raw, X_test_raw, y_train_raw, y_test_raw = train_test_split(
    X_raw, y_raw,
    test_size=0.2,
    stratify=y_raw,          # guarantees class proportions in both sets
    shuffle=True,
    random_state=42
)

# Step C: SEQUENCE CREATION (POST-SPLIT)
# Train sequences contain ONLY train rows; test sequences contain ONLY test rows.
X_train_seq, y_train_seq = create_sequences_optimized(
    X_train_arr, y_train_arr, window_size=10, stride=1
)
X_test_seq, y_test_seq = create_sequences_optimized(
    X_test_arr, y_test_arr, window_size=10, stride=1
)

The create_sequences_optimized function uses np.lib.stride_tricks.sliding_window_view for memory-efficient windowing, producing a view rather than a full copy of the underlying array until materialisation. For a file with N valid raw flows after splitting, this produces ⌊(N − 10)/1⌋ + 1 sequences per partition. The resulting 3D arrays are stored as .npy files in data/processed_lstm/: X_train.npy (shape: N_train × 10 × 20) and X_test.npy (shape: N_test × 10 × 20).

Leakage-free scaling for the 3D case requires an additional reshape step: the 3D array is flattened to 2D (N × window_size, features) = (N × 10, 20) before MinMaxScaler.fit_transform is applied on train data only, and then reshaped back to 3D. This ensures that the scaler observes only the range of feature values present in training sequences, not in test sequences.

C. Class Weights for Imbalance Compensation

Both preprocessing pipelines compute and persist inverse-frequency class weights derived from the training set label distribution:

Class Weight Formula:    wc = Ntotal / (K × Nc)    where K = number of classes (3), Nc = training samples in class c

This formula assigns higher weight to minority classes (Semantic: weight ≈ 4–5×) and lower weight to the majority class (Benign: weight ≈ 0.4×), effectively up-weighting the loss contribution of underrepresented attack types during training. For ML models this is provided as class_weight='balanced' (scikit-learn) or compute_sample_weight('balanced') (XGBoost). For DL models the computed weights are loaded from models/class_weights.json and passed as class_weight to model.fit().

Section 3 Summary — Key Preprocessing Decisions

Dataset: CICIDS2017 — 2.83M flows, 8 per-day source files, 78 CICFlowMeter features. Selected for multi-day temporal diversity, realistic benign traffic, and heterogeneous attack coverage.

Taxonomy: 3-class (Benign=0 / Volumetric=1 / Semantic=2) via classes_map.json, motivated by SOC triage routing and class cardinality requirements. Label normalisation (NFKC + whitespace collapsing) handles CICIDS2017 formatting inconsistencies.

Features: Top-20 CICFlowMeter features selected by XGBoost importance + RF Gini decrease consensus. Dominated by backward-direction packet statistics (8/20 features) and IAT timing descriptors (3/20). All ±inf / NaN values replaced by 0 before scaling.

Split Strategy (C1): Per-file stratified 80/10/10 split (ML) or 80/20 split (LSTM), applied independently to each source file before cross-file aggregation. Eliminates temporal leakage and guarantees minority class representation in every partition.

Scaling: MinMaxScaler fitted on training partition only, applied as a fixed transformation to validation and test partitions. Scaler serialised for production deployment.
