Section 5

Experimental Setup, Infrastructure & AutoML Benchmarking

This section documents the complete experimental environment: the hardware and software infrastructure on which all training and evaluation was conducted (§5.1), the evaluation metric suite and latency measurement methodology (§5.2), the AutoML benchmarking procedure using PyCaret (§5.3), and an overview of the live production architecture that motivates the latency constraints imposed on model selection (§5.4). Together, these subsections provide the reproducibility context required to re-implement, verify, and extend the results reported in §8.

5.1

Computational Environment

A. Hardware Configuration

All training, hyperparameter search, and evaluation experiments were conducted on a single workstation. Deep learning models (LSTM, BiLSTM) leveraged CUDA-accelerated training with TensorFlow memory growth enabled to prevent GPU out-of-memory errors during 1.8 GB sequence data processing. XGBoost training used CUDA via the tree_method='hist' + device='cuda' backend (XGBoost ≥ 2.0). Traditional ML models (Random Forest, Decision Tree) ran on CPU with multi-core parallelism (n_jobs=4).

⚠ Memory Engineering Note

The full 3D LSTM training corpus (X_train shape: approx. (2,265,000, 10, 20), float32) occupies approximately 1.8 GB on disk. Loading this directly into GPU memory would cause OOM errors. The training pipeline mitigates this through a memory-mapped generator (np.load(..., mmap_mode='r')) streamed via tf.data.Dataset.from_generator() → batch(256) → prefetch(AUTOTUNE). This keeps only one batch (~4 MB) in GPU memory at any time, enabling full-dataset training on consumer-grade hardware.

5.2

Evaluation Metrics & Measurement Methodology

A. Classification Metrics

All five models are evaluated using an identical, pre-specified metric suite to enable direct cross-model comparison. All metrics are computed on the held-out test partition produced by the per-file stratified split (§3.4). Macro-averaged metrics are selected as the primary reporting standard because they give equal weight to each class regardless of support, making them sensitive to poor performance on the minority Semantic class (~6%):

Table 11. Evaluation Metric Suite — Definitions and Rationale
Metric Averaging Formula (per-class) Role in this Study
F1-Score Macro 2·P·R / (P+R) Primary metric — balances P/R, treats all classes equally
Precision Macro TP / (TP+FP) Measures false alarm rate; critical for SOC alert fatigue
Recall Macro TP / (TP+FN) Measures attack detection rate; critical for security coverage
Accuracy Overall Σ TP / N Reported for comparison with prior work (biased by class imbalance)
Weighted F1 Weighted Support-weighted mean Reported for completeness; not primary due to imbalance bias
OvR ROC-AUC Macro One-vs-Rest per class Threshold-independent discriminative power; uses softprob output
3×3 Confusion Matrix — Raw counts Reveals cross-class confusion (Volumetric vs. Semantic misclassification)
Inference Latency — ms / sample Production viability — measured on 10,000-sample benchmark batch
Macro-averaging computes the metric independently for each class and takes the unweighted mean, giving Benign (80%), Volumetric (14%), and Semantic (6%) equal influence on the final score. This is the standard recommended by the ML community for imbalanced multi-class evaluation.
B. Inference Latency Measurement

Inference latency is a production-critical metric for any NIDS system: a model that achieves 99% accuracy but requires 500 ms per flow cannot operate in real-time. The measurement protocol (from stress_test.py and the evaluate scripts) benchmarks on a fixed 10,000-sample batch to average out OS scheduling jitter:

Python — latency measurement protocol
# Consistent across all five model evaluation scripts
sample_size = 10_000          # fixed benchmark size
X_sample = X_test.iloc[:sample_size]

start = time.time()
_ = model.predict_proba(X_sample)  # or model.predict()
elapsed = time.time() - start

latency_ms  = (elapsed / sample_size) * 1000   # ms per sample
throughput  = sample_size / elapsed            # samples / second

# Thresholds (from stress_test.py):
# > 10,000 samples/s → EXCELLENT (live stream capable)
# > 2,000  samples/s → GOOD     (normal traffic)

5.3

AutoML Benchmarking with PyCaret

In addition to the five manually designed and tuned models (§4), we conducted an AutoML benchmarking experiment using PyCaret 3.x (classification module) to produce a comprehensive, automated baseline across 10+ algorithm families simultaneously. The AutoML results serve two purposes: (1) they identify whether the hand-tuned models are competitive with the top of the automated leaderboard, and (2) they provide a reference point for practitioners who wish to evaluate the dataset with minimal engineering effort.

A. PyCaret Setup and Configuration

The PyCaret experiment (experiments/pycaret_setup.ipynb) was configured to use the same 3-class training data produced by the per-file stratified split pipeline. Key setup parameters include:

Python — experiments/pycaret_setup.ipynb 
from pycaret.classification import *

# Setup: 3-class classification on preprocessed training data
s = setup(
    data      = train_df,         # from data/processed_ml/train.csv
    target    = 'Label',          # integer labels: 0, 1, 2
    session_id = 42,              # reproducibility seed
    fold      = 5,               # 5-fold stratified CV
    metric    = 'F1',            # primary optimisation metric (macro)
    verbose   = True
)

# Compare all available algorithms — ranked by Macro F1
best_models = compare_models(sort='F1', n_select=5)
B. Algorithms Evaluated

PyCaret's compare_models() routine benchmarks all available classification algorithms in a single call, applying each with default hyperparameters and 5-fold stratified cross-validation scored on Macro F1. The algorithms evaluated include:

C. Champion Model and Artefact Persistence

Following completion of the leaderboard comparison, the top-ranked model (the PyCaret champion) was serialised using PyCaret's native save_model() and persisted to models/pycaret_champion.pkl. This artefact enables the production Live Bridge (§5.4) to optionally load and invoke the PyCaret champion at runtime — all five trained models plus the PyCaret champion are hot-swappable within the inference pipeline without restarting the server.

ℹ Role of AutoML in This Study

The PyCaret benchmarking is not intended to produce the final production model. Rather, it serves as an automated breadth-first sweep across the algorithm space: if the top AutoML leaderboard entry significantly outperforms our hand-tuned models, that signals a gap in manual tuning. If the hand-tuned models outperform or match the AutoML champion, it validates that the hyperparameter search effort (§4.1–§4.3) and training protocol (GPU acceleration, custom imbalance handling) were effective. The comparison is documented in §8.4.

5.4

Production Architecture Overview — Live Bridge Mechanism

The experimental models described in §4 were designed with production deployment as an explicit constraint. This section provides an overview of the Live Bridge Mechanism — the real-time inference pipeline that connects raw network traffic to model predictions — to contextualise the latency and throughput requirements that motivated the model design choices. A detailed description of the full production architecture is presented in §6.

A. Pipeline Architecture

The Live Bridge implements a five-stage streaming pipeline (src/live_bridge.py):

Network Interface Scapy sniff() · Wi-Fi/ETH
→
PCAP Buffer temp_live.pcap · 60s window
→
CICFlowMeter CLI 78 bidirectional features
→
Kafka Producer Topic: network-traffic · 127.0.0.1:9092
→
AI Consumer Model inference · 3-class prediction
B. Feature Alignment Bridge

A key engineering challenge in the production pipeline is the column naming mismatch between CICFlowMeter's output (snake_case, e.g. fwd_pkt_len_max) and the training data schema (Title Case, e.g. Fwd Packet Length Max). The Live Bridge implements a deterministic column rename map (COLUMN_RENAME_MAP in live_bridge.py, lines 389–466) covering all 78 CICFlowMeter feature names, followed by reindex(columns=EXPECTED_FEATURES, fill_value=0) to handle any residual missing columns. Additionally, ±inf and NaN values from the live feature extractor are replaced with 0 — exactly mirroring the preprocessing applied during offline training (§3.3).

Live Bridge Feature Alignment Protocol — live_bridge.py
Step 1: df.columns.str.strip() # remove leading/trailing whitespace
Step 2: df.drop(DROP_COLS, ...) # remove metadata (IP, port, timestamp, labels)
Step 3: df.rename(COLUMN_RENAME_MAP) # snake_case → Title Case (78 mappings)
Step 4: df["Fwd Header Length.1"] = ... # synthesise duplicate column
Step 5: df.reindex(EXPECTED_FEATURES, fill_value=0) # align to Top-20 subset
Step 6: df.replace([±inf], 0).fillna(0) # identical to offline preprocessing
C. Kafka Integration

The Live Bridge uses Confluent Kafka as the message bus between the packet capture process and the AI consumer (inference) process. Each extracted flow is serialised as a JSON message (including the 78-feature vector plus metadata: timestamp, src/dst IP) and published to the network-traffic topic on the local broker (127.0.0.1:9092). The Kafka producer is configured with acks='all', retries=3, and max.in.flight.requests.per.connection=1 to guarantee delivery ordering and prevent message reordering under rare network conditions. A background TrafficLogger thread asynchronously archives all flows to data/live_captured_traffic.csv via a 25-row buffer with a 30-second flush interval, enabling post-hoc analysis and model retraining on captured live traffic.

D. Implication for Model Selection

The production pipeline imposes a hard latency ceiling on model selection. Flows are extracted in overlapping 60-second windows by CICFlowMeter; the AI consumer must classify all flows within that window before the next window arrives. This constrains the acceptable per-sample inference latency to well under 1 ms for any model intended for production deployment. This constraint directly motivates the latency emphasis in §5.2 and the discussion of accuracy-versus-latency trade-offs in §8.3.

✅ Design Principle — Same Features, Same Transform

The Live Bridge applies an identical feature transformation sequence to live traffic as was applied during offline preprocessing (§3.3): the same 78 feature names, the same ±inf/NaN → 0 substitution, and the same MinMaxScaler (loaded from models/scaler.pkl, fitted on training data only in §3.4). This one-to-one correspondence between the offline preprocessing pipeline and the production serving pipeline is a necessary condition for valid real-time inference and is enforced at code level through shared constants and serialised scaler artefacts.

Section 5 Summary — Experimental Infrastructure

Hardware: CUDA GPU + multi-core CPU; TensorFlow memory growth enabled; mmap-mode streaming for 1.8 GB LSTM corpus (batch=256, prefetch=AUTOTUNE).

Metrics: Primary = Macro F1; Secondary = Macro P/R, Accuracy, Weighted F1, OvR ROC-AUC per class. Latency: 10K-sample benchmark giving ms/sample and samples/sec throughput. Threshold: >10K samples/sec = live-stream viable.

AutoML (PyCaret): 13+ algorithm families benchmarked via 5-fold stratified CV, sorted by Macro F1. Champion saved to models/pycaret_champion.pkl — hot-swappable in production inference. Serves as an automated reference baseline for validating hand-tuned model effectiveness (§8.4).

Live Bridge: Scapy → PCAP → CICFlowMeter CLI (78 features) → Kafka (acks='all', network-traffic topic) → AI consumer. Column rename map (78 entries) bridges CICFlowMeter snake_case to training Title Case schema. Async TrafficLogger archives live flows for future retraining.
