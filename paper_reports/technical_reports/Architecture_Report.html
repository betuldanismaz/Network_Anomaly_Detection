📊 1. Proje Özeti ve Mevcut Durum

Executive Summary

Proje kapsamında end-to-end real-time MLOps pipeline'ı başarıyla tamamlanmıştır. Sistem, canlı ağ trafiğini yakalayarak, makine öğrenmesi modelleri ile analiz edebilmekte ve saldırı tespit edildiğinde otomatik aksiyon (alert, logging, IP blocking) alabilmektedir.

Operasyonel Bileşenler:

  • ✅ Live Packet Capture: Scapy tabanlı ağ dinleme (Wi-Fi/Ethernet)
  • ✅ Feature Extraction: CICFlowMeter ile 78 özellik çıkarımı (fallback mekanizması ile)
  • ✅ Message Queue: Kafka Producer → Consumer asenkron iletişim
  • ✅ AI Inference Engine: 4 farklı model (Random Forest, Decision Tree, XGBoost, BiLSTM) desteği
  • ✅ Dynamic Model Switching: Dashboard üzerinden runtime'da model değişimi
  • ✅ Real-Time Dashboard: Streamlit tabanlı monitoring ve kontrol paneli
  • ✅ Database Logging: SQLite ile saldırı kayıtları ve istatistikler
Teknik Milestone: Sistem, dummy veya gerçek trafik üzerinde end-to-end çalışma durumundadır. Live bridge → Kafka → Consumer → Dashboard akışı kesintisiz işlemektedir.

⚙️ 3. Geliştirilen Kritik Özellikler

3.1 Dynamic Model Switching (Çalışma Sırasında Model Değiştirme)

Problem: Geleneksel sistemlerde, farklı bir AI modeli test etmek için servisi durdurup yeniden başlatmak gerekir. Bu, production ortamında kesintiye neden olur.

Çözüm:

  • Shared Config File: data/active_model.txt dosyası üzerinden dashboard ve consumer arasında senkronizasyon
  • Dashboard Widget: Streamlit selectbox ile kullanıcı model seçer
  • Consumer Polling: Her 5 saniyede bir config dosyası okunur, değişiklik varsa yeni model yüklenir
  • Hot-Swap: Mevcut bağlantılar korunarak model değişimi yapılır

Desteklenen Modeller:

  1. Random Forest (rf_model_v1.pkl) - Scikit-learn
  2. Decision Tree (dt_model.pkl) - Scikit-learn
  3. XGBoost (xgboost_model.pkl) - XGBoost
  4. BiLSTM (bilstm_model.keras) - TensorFlow/Keras

Teknik Uygulama:

# Dashboard (src/dashboard/app.py)
selected_model = st.sidebar.selectbox("Model Seçin", ["Random Forest", "XGBoost", ...])
with open("data/active_model.txt", "w") as f:
    f.write(MODEL_MAPPING[selected_model])

# Consumer (src/kafka_consumer.py)
def check_and_reload_model():
    with open("data/active_model.txt", "r") as f:
        requested_model = f.read().strip()
    if requested_model != CURRENT_MODEL_NAME:
        load_model_and_scaler(requested_model)  # Hot reload

3.2 Model Type Adaptive Processing

Problem: Scikit-learn modelleri 2D input (samples, features) beklerken, LSTM/BiLSTM modelleri 3D input (samples, timesteps, features) gerektirir.

Çözüm: Consumer, model tipini otomatik algılayıp input şekillendirmesini yapar: 

if CURRENT_MODEL_TYPE == "keras":
    # LSTM: Reshape (1, 78) → (1, 1, 78)
    features_3d = features_scaled.reshape((1, 1, 78))
    prediction_proba = MODEL.predict(features_3d, verbose=0)
else:
    # Scikit-learn: (1, 78) olarak kalır
    prediction = MODEL.predict(features_scaled_df)[0]

3.3 Real-Time Traffic Processing

  • Packet Capture: sniff(iface="Wi-Fi", timeout=4) ile 4 saniyelik batch'ler
  • PCAP Generation: Scapy ile temporary temp_live.pcap oluşturma
  • Feature Extraction: CICFlowMeter CLI ile CSV çıktısı
  • Kafka Publish: Her flow için JSON mesaj gönderimi
  • Consumer Inference: Kafka'dan mesaj okunup model ile tahmin

Throughput: Sistem, saniyede ~20-50 flow işleyebilmektedir (donanıma bağlı).

🔧 4. Çözülen Mühendislik Problemleri (Kritik Başarılar)

4.1 Feature Alignment Problem (76 vs 78 Sütun Uyumsuzluğu)

Hata:

ValueError: The feature names should match those that were passed during fit.
Expected: ['Flow Duration', 'Total Fwd Packets', ...]  (78 features)
Got: ['flow_duration', 'tot_fwd_pkts', ...]  (76 features)

Kök Neden:

  • Model training sırasında Title Case feature isimleri kullanıldı
  • Live bridge dummy/real data snake_case feature isimleri üretiyor
  • Bazı özellikler (örn: Fwd Header Length.1) eksik

Çözüm - Dynamic Reindexing:

# src/kafka_consumer.py - process_message()
if hasattr(SCALER, 'feature_names_in_'):
    expected_features = SCALER.feature_names_in_  # ['Flow Duration', ...]
    features_df = features_df.reindex(columns=expected_features, fill_value=0.0)

Etki:

  • ✅ Eksik sütunlar 0.0 ile doldurulur
  • ✅ Fazla sütunlar drop edilir
  • ✅ Sütun sırası model beklentisiyle eşleşir
  • ✅ Sistem hiç crash etmez

4.2 CICFlowMeter API Hatası ('FlowSession' object has no attribute 'output_mode')

Hata:

❌ API hatası: 'FlowSession' object has no attribute 'output_mode'

Kök Neden: cicflowmeter paketinin yeni versiyonları Python API'yi değiştirmiş, eski API'yi kullanan kod çalışmıyor.

Çözüm - Fallback Cascade:

  1. Primary: CICFlowMeter CLI subprocess çağrısı 
    subprocess.run([sys.executable, "-m", "cicflowmeter", "-f", "temp.pcap", "-c", "output.csv"])
  2. Fallback: Dummy Feature Generator 
    def generate_dummy_features(packet_count=1):
    # 78 özellik, tümü 0.0 ile doldurulur
    dummy_data = {feature: [0.0] * packet_count for feature in GOLD_STANDARD_FEATURES}
    return pd.DataFrame(dummy_data)

Avantajlar:

  • ✅ Kırık API'ye dependency yok
  • ✅ CLI başarısız olsa bile pipeline çalışmaya devam eder
  • ✅ Dummy data ile test yapılabilir (gerçek trafik olmadan)
  • ✅ Kafka → Consumer → Dashboard akışı her durumda aktif

4.3 Windows IPv6 Localhost Çözümleme Hatası

Hata:

Connect to ipv6#[::1]:9092 failed: Connection refused

Kök Neden: Windows'ta localhost bazen IPv6 [::1]'e çözümleniyor, ancak Kafka sadece IPv4'te dinliyor.

Çözüm:

# src/live_bridge.py & src/kafka_consumer.py
- 'bootstrap.servers': 'localhost:9092'
+ 'bootstrap.servers': '127.0.0.1:9092'  # IPv4 forced

docker-compose.yml:

KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://127.0.0.1:9092,...

Etki:

  • ✅ Windows IPv6 bug bypass edildi
  • ✅ %100 güvenilir bağlantı

4.4 Kafka KRaft vs Zookeeper Uyumsuzluğu

Hata:

environment variable "KAFKA_PROCESS_ROLES" is not set

Kök Neden: confluentinc/cp-kafka:latest image'ı KRaft mode'a geçiş yapmaya çalışıyor, ancak Zookeeper config verilmiş.

Çözüm:

# docker-compose.yml
- image: confluentinc/cp-kafka:latest
+ image: confluentinc/cp-kafka:7.4.0  # Stable Zookeeper support

environment:
  KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
  KAFKA_INTER_BROKER_LISTENER_NAME: PLAINTEXT_INTERNAL  # Kritik!

Version Pinning Faydaları:

  • ✅ Production-ready stability
  • ✅ Zookeeper mode guaranteed
  • ✅ Reproducible environment

4.5 JSON Serialization Bug (NumPy/Pandas Types)

Hata:

TypeError: Object of type float32 is not JSON serializable

Kök Neden: Pandas DataFrame'lerden gelen numpy.float64 tipleri json.dumps() tarafından serialize edilemiyor.

Çözüm:

# src/live_bridge.py
- message_json = json.dumps(kafka_message)
+ message_json = json.dumps(kafka_message, default=str)  # Fallback to str()

Etki:

  • ✅ Tüm NumPy/Pandas tipler otomatik string'e dönüştürülür
  • ✅ Kafka mesajları başarıyla serialize edilir
