Section 6

Real-Time Production Architecture

A primary distinguishing contribution of this work relative to prior NIDS literature is the implementation of a production-grade inference pipeline that bridges the offline benchmark experiment (§§3–5) with live network traffic. This section describes, in engineering detail, the end-to-end system architecture that enables real-time packet capture, flow feature extraction, model inference, alert generation, and optional automated response — all running concurrently as distinct, loosely-coupled microservices connected via Apache Kafka. We additionally describe the model hot-swap mechanism (§6.3) that enables runtime model selection without restarting any process, and the integrated firewall response module (§6.4) that enables closed-loop automated IP blocking upon confirmed attack detection.

6.1

System Architecture Overview

The production system is structured as a producer–consumer architecture over an Apache Kafka message bus, separating the computationally heavy packet capture and feature extraction stage (CPU-bound, blocking) from the model inference and logging stage (GPU/CPU, asynchronous). This decoupling ensures that a slow CICFlowMeter extraction cycle does not block alert generation, and that multiple consumer instances can be scaled independently of the capture process.

6.2

Live Bridge Mechanism — Key Engineering Contribution

The Live Bridge (src/live_bridge.py, 863 lines) is the core production engineering contribution of this work. It implements the complete capture-to-Kafka pipeline as a single continuous process, handling the inherent messiness of real-world packet capture: incomplete flows, feature extractor failures, column name mismatches, and ±∞/NaN artefacts.

A. Packet Capture Loop

The capture loop uses Scapy's sniff() with a configurable network interface (read from the NETWORK_INTERFACE environment variable, default: "Wi-Fi") and a 4-second window. All captured packets are written to a temporary PCAP buffer (temp_live.pcap) using wrpcap(), then CICFlowMeter is invoked as a subprocess to extract bidirectional flow features. The main loop runs indefinitely until CTRL+C, with a 10-iteration console status update:

Python — live_bridge.py (lines 834–848)
while True:
    packets = sniff(iface=INTERFACE, timeout=4)   # 4-second capture window
    if len(packets) > 0:
        wrpcap(TEMP_PCAP, packets)             # persist to temp_live.pcap
        feature_extraction_and_predict()      # CICFlowMeter → Kafka publish
        iteration_count += 1
    if iteration_count % 10 == 0:
        print(f"📊 Producer Stats: {iteration_count} batches sent")
B. CICFlowMeter CLI Integration & Fallback

CICFlowMeter is invoked via subprocess.run() rather than the Python API to avoid version compatibility issues on Windows. Two invocation strategies are attempted in sequence: (1) python -m cicflowmeter and, if that fails, (2) the global cicflowmeter CLI command. CICFlowMeter occasionally produces output under a modified filename (e.g., temp_live_Flow.csv); the bridge handles this with an automatic file rename detection. If extraction fails entirely after both attempts, a dummy zero-filled feature row is generated to keep the Kafka pipeline alive for monitoring — preventing a single packet capture failure from silently stalling the entire inference chain.

C. Kafka Publish — Message Schema

Each extracted flow is serialised as a JSON message and published to the network-traffic Kafka topic. The message schema includes both inference-relevant features and metadata for downstream logging:

Kafka Message Schema — live_bridge.py (lines 553–563)
{
"timestamp": "2026-03-05T16:34:02.123456", // ISO-8601
"src_ip": "192.168.1.105",
"dst_ip": "142.250.74.46",
"features": { "flow_duration": 0.045, "fwd_pkt_len_max": 1460, ... }, // 78 features
"feature_count": 78,
"producer_id": "live_bridge_v1"
}

The Kafka producer is configured with acks='all' (leader + all replicas must acknowledge), retries=3, and max.in.flight.requests.per.connection=1 to guarantee at-least-once message delivery with preserved ordering.

6.3

AI Consumer & Runtime Model Hot-Swap

The AI Consumer (src/kafka_consumer.py, 508 lines) subscribes to the network-traffic Kafka topic and processes each flow message through a complete inference pipeline: JSON parsing → feature alignment → MinMaxScaler transform → model forward pass → class prediction → logging.

A. Dual-Format Model Loading

A key engineering feature of the consumer is its ability to load and serve any of the five trained models — or the PyCaret champion — without code changes, through a single unified loader:

Table 13. Consumer Model Loader — Supported Formats and Inference Paths
File Extension Model Type Loader Input Shape Scaler Used
.pkl XGBoost / RF / DT / PyCaret joblib.load() (1, 78) scaler.pkl
.keras LSTM / BiLSTM (Keras) keras.models.load_model() (1, 1, 78) reshaped scaler_lstm.pkl
.h5 Legacy Keras format keras.models.load_model() (1, 1, 78) reshaped scaler_lstm.pkl
The consumer automatically detects model type from the file extension and applies the appropriate scaler and input reshape. For Keras models, the single-flow feature vector is reshaped from (1, 78) → (1, 1, 78) to satisfy the expected 3D LSTM input format (batch × timesteps × features).
B. Zero-Downtime Model Hot-Swap

The consumer implements a polling-based hot-swap mechanism that checks for a model change request every 5 seconds (configurable via CONFIG_CHECK_INTERVAL) by reading a single-line text file (data/active_model.txt). This file can be updated by the web dashboard, an operations script, or any external process. When a change is detected, the consumer reloads the new model in-place — no consumer restart or Kafka consumer group rebalancing is required:

Python — kafka_consumer.py (lines 405–438) 
def check_and_reload_model():
    # Runs every 5 seconds in the main polling loop
    with open(ACTIVE_MODEL_CONFIG) as f:
        requested_model = f.read().strip()   # e.g., "lstm_best.keras"

    if requested_model != CURRENT_MODEL_NAME:
        print(f"🔄 MODEL SWITCH: {CURRENT_MODEL_NAME} → {requested_model}")
        load_model_and_scaler(requested_model)   # hot reload
        # No restart — next poll immediately uses new model
ℹ Operational Significance of Hot-Swap

Hot-swap enables a critical operational workflow: an analyst can evaluate all five models against live traffic without interrupting the capture pipeline. The model name written to active_model.txt controls which trained model services the next Kafka message. This allows rapid A/B comparison of, for example, XGBoost (low latency) versus BiLSTM (higher sequential context) on the same live traffic stream — a comparison not possible in offline evaluation.

6.4

Automated Firewall Response & IP Whitelisting

When a flow is classified as an attack (class 1 or 2), the consumer optionally triggers an automated firewall block via src/utils/firewall_manager.py. The IP blocking is cross-platform — it uses netsh advfirewall on Windows and iptables on Linux — and includes duplicate-rule checking to prevent accumulation of redundant rules:

🔴 Windows Firewall
netsh advfirewall
  • Rule name: Block_AI_{src_ip}
  • Direction: inbound (dir=in)
  • Action: action=block remoteip={ip}
  • Duplicate check before adding
  • unblock_ip() → removes rule by name
🟠 Linux IPTables
iptables -A INPUT
  • Rule: -s {ip} -j DROP
  • Pre-check: iptables -C INPUT
  • Removal: iptables -D INPUT
  • Platform auto-detection via platform.system()
🔵 IP Whitelist Guard
WHITELIST_IPS env var
  • Default: 127.0.0.1, 192.168.1.1, localhost
  • Configurable via .env file
  • Even if attack is detected, whitelisted IPs are never blocked
  • Alert still logged with "ALLOWED" status
🟢 SQLite Alert Log
utils/db_manager.py → alerts.db
  • log_attack(src_ip, action, reason)
  • Actions: BLOCKED / ALLOWED
  • Persistent alert history for post-incident analysis
  • Readable by dashboard (src/dashboard/app.py)
⚠ Auto-Block is Opt-In by Design

The automated block_ip(src_ip) call in kafka_consumer.py (line 345) is intentionally commented out in the delivered codebase. The system logs and alerts on all detections but requires explicit operator action to enable automated blocking. This design decision reflects the security-engineering principle that automated defensive actions carry operational risk (e.g., blocking legitimate traffic on false positives), and should only be enabled after threshold validation on a per-deployment basis.

6.5

Attack Simulation Environment

A critical gap in most NIDS research is the inability to validate model performance beyond the offline benchmark: the model may achieve high F1 on CICIDS2017 test data but fail on live traffic with different packet timing, fragmentation patterns, or flow completion rates. To bridge this gap, the production architecture is designed to support closed-loop dynamic validation using a co-located attack simulation environment.

A. Simulation Methodology

The attack simulation environment enables controlled injection of both Volumetric and Semantic attack patterns against the Live Bridge while it is actively monitoring the same network interface. This creates a closed validation loop: attack tool → network interface → Scapy sniff → CICFlowMeter → Kafka → AI Consumer → detection result. The detection outcome (class 0/1/2) is compared against the known ground-truth attack type to compute live detection rates.

Table 14. Attack Simulation Environment — Attack Types, Tools, and CICIDS2017 Equivalence
Attack Category Class (Taxonomy) Simulation Tool Protocol CICIDS2017 Equivalent Expected Detection
UDP Flood 1 — Volumetric hping3 --udp UDP DDoS LOIT / UDP Flood High
SYN Flood 1 — Volumetric hping3 -S --flood TCP SYN DoS Hulk / GoldenEye High
ICMP Flood 1 — Volumetric hping3 --icmp ICMP DoS Heartbleed High
Slowloris 1 — Volumetric slowloris (Python) HTTP (slow) DoS Slowloris / SlowHTTPTest Moderate–High
Port Scan 2 — Semantic nmap -sS / -sU TCP/UDP PortScan High
SSH Brute Force 2 — Semantic hydra / medusa SSH Brute Force SSH Moderate

B. Validation Metrics for Live Testing

The live attack simulation environment measures three complementary quantities not captured by the offline benchmark:

✅ Contribution C4 — Closed-Loop Dynamic Validation

The combination of the Live Bridge (§6.2), hot-swap model selection (§6.3), and the attack simulation environment establishes a closed-loop validation methodology that is absent from virtually all prior NIDS benchmark studies. Rather than reporting offline test-set accuracy as a proxy for production performance, this system enables direct measurement of alert latency, live false positive rates, and per-attack detection rates under controlled but realistic network conditions. The simulation results are reported in §8.5.

6.6

Continuous Traffic Harvesting for Model Retraining

The production pipeline includes an asynchronous TrafficLogger (live_bridge.py, class TrafficLogger) that persists every processed flow — 78 raw features, predicted label, confidence score, and timestamp — to a live capture CSV (data/live_captured_traffic.csv). This data collection mechanism supports continual model improvement: as live traffic diverges from the CICIDS2017 training distribution (due to network topology changes, new application protocols, or evolving attack patterns), the harvested flows can be labelled by an analyst and used to fine-tune or retrain the deployed models.

The TrafficLogger runs on a dedicated background writer thread to minimise impact on the inference latency of the main capture loop. Data is buffered in memory (25 rows) and flushed to disk every 30 seconds, or when the buffer fills, using append-mode CSV writes. A graceful shutdown hook registered via atexit.register(self.shutdown) ensures that all buffered rows are flushed to disk even on abnormal process termination.

Section 6 Summary — Real-Time Production Architecture

15-component pipeline: Scapy (4s window) → PCAP → CICFlowMeter CLI (subprocess, 90s timeout, fallback to dummy) → JSON Kafka message → network-traffic topic → AI Consumer → MinMaxScaler transform → active model → 3-class prediction → CSV log + SQLite alerts.

Hot-swap: Consumer polls data/active_model.txt every 5 seconds. Any of 6 models (XGBoost, RF, DT, LSTM, BiLSTM, PyCaret champion) switchable at runtime — zero downtime, no restart.

Automated response: firewall_manager.py provides cross-platform IP blocking (netsh/iptables), whitelist guard, and deduplication. Auto-block is opt-in by default.

Attack simulation: 8 attack scenarios (3 Volumetric + 4 Semantic + benign) via hping3, nmap, hydra, slowloris, nikto. Enables closed-loop live validation: E2E latency, live FPR, per-attack detection rate.

Retraining pipeline: Async TrafficLogger (25-row buffer, 30s flush) archives all live flows (78 features + prediction + confidence) for future fine-tuning.
