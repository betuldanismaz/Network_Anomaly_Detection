Section 7

MLOps & Engineering Practices

Academic NIDS research commonly ends at the offline evaluation table: models are trained, evaluated, and reported without addressing reproducibility, artifact management, data integrity verification, or the operational tooling required to sustain a deployed system. This section documents the end-to-end MLOps practices embedded in this project — spanning reproducibility infrastructure (§7.1), a multi-format model artifact registry (§7.2), automated data integrity auditing (§7.3), explainability via SHAP (§7.4), containerised infrastructure (§7.5), and automated reporting and observability (§7.6). These practices collectively address the reproducibility crisis identified in §2.5 and position this work for direct extension by future researchers.

7.1

Reproducibility Infrastructure

Reproducibility of ML experiments in security research is notoriously difficult due to stochastic training algorithms, time-varying data sources, and opaque hyperparameter choices. This project enforces reproducibility at four distinct levels:

Table 16. Reproducibility Controls Applied at Each Pipeline Stage
Stage Reproducibility Control Implementation
Data Splitting Fixed random seed + deterministic per-file stratified split random_state=42 in all train_test_split() calls
Scaling Leakage-free scaler: fitted on train only, serialised immediately joblib.dump(scaler, 'models/scaler.pkl') after fit
Traditional ML All sklearn & XGBoost estimators seeded random_state=42 in every constructor call
Deep Learning TensorFlow global seed + NumPy seed + Python hash seed tf.random.set_seed(42), numpy.random.seed(42), PYTHONHASHSEED='42'
AutoML (PyCaret) Session ID propagated to all internal fold splits session_id=42 in setup()
Feature Selection Centralised, immutable Top-20 list in version-controlled config src/config.py → TOP_FEATURES imported by all scripts
Dependency Pinning All runtime dependencies declared without wildcards requirements.txt with 13 pinned packages
Seed value 42 is applied universally as the first operation in every training script, before any library import that could trigger stochastic initialisation at load time. This ordering prevents subtle non-determinism from implicit state set during import.

Model Artifact Registry & Versioned Artefacts

Each training run produces a structured set of artefacts written to the models/ and reports/ directories. This predictable naming convention enables the Kafka consumer's hot-swap mechanism (§6.3) to discover and load any model by filename without code changes, and enables analysts to audit any model's training configuration by inspecting the paired .json config file.

🔑 Design Principle — Config-Paired Artefacts

Every .pkl / .keras model file is accompanied by a .json configuration file that records the exact hyperparameters used to produce the model (including the grid-search winner for RF/DT). This pairing ensures that the model binary is never orphaned from its training context — a researcher loading the model 6 months later can immediately inspect the configuration without reading source code.

7.3

Automated Data Integrity Auditing

The preprocessing pipeline is followed by an automated 8-check data integrity audit implemented in src/utils/data_audit.py (175 lines). The audit must pass before any model training commences, serving as a pre-training gate that prevents silent data quality issues from propagating to reported metrics.

Check 1
Feature Count Verification
Asserts exactly 20 features + 1 label column across all three splits. Guards against accidental column retention after feature selection.
✅ Expected: 21 columns per split
Check 2
Data Leakage Detection
Performs an inner join between train and test feature frames. Any exact-duplicate rows indicate temporal leakage from the split procedure.
✅ Expected: 0 overlapping rows
Check 3
Forbidden Identifier Check
Scans for columns that should have been dropped: Flow ID, Source IP, Source Port, Destination IP, Timestamp, Date.
✅ Expected: 0 forbidden columns
Check 4
NaN & Infinity Sanity Check
Confirms all ±inf values and NaNs were replaced with 0 during preprocessing. Any residuals would produce undefined model behaviour.
✅ Expected: 0 NaNs, 0 Infs
Check 5
Class Distribution Stratification
Verifies that class proportions in train/val/test are statistically consistent with the full dataset distribution (per-file stratification guarantee).
✅ Expected: ≈80/14/6% across all splits
Check 6
Scaling Verification
Checks that all feature values in the training set lie within [−0.01, 1.01]. Values outside this range indicate the MinMaxScaler was not applied correctly.
✅ Expected: all values ∈ [0, 1]
Check 7
Data Type Efficiency
Flags float64 columns and recommends float32 downcast to reduce in-memory size by 50% — relevant for 2.83M-row datasets.
✅ Notice level — not a blocking error
Check 8
Summary Report
Prints total row counts, split ratios, and feature count as an automated confirmation that the preprocessing pipeline produced the expected dataset dimensions.
✅ N_train / N_val / N_test / K_features

The LSTM-specific audit (src/utils/data_audit_lstm.py) applies an analogous suite to the 3D numpy arrays, additionally verifying array shapes ((N, 10, 20)), dtype (float32), value range ([0, 1]), and the absence of the sequence-creation data leakage pattern — confirming that the "Split Raw First" strategy (§3.5) was correctly applied before sequence generation.

7.4

Explainability Engine — SHAP Feature Attribution

Real-world Security Operations Centre (SOC) deployments require not only that a model detected an attack, but why — an analyst acting on an alert needs to understand which specific flow features triggered the detection in order to correlate with other signals, investigate potential false positives, or prepare an incident report. To address this, the system includes a production-grade SHAP-based explainability engine (src/utils/xai_engine.py) integrated into the Live Bridge inference loop.

A. SHAP Explainer Generation

The SHAP explainer is generated offline by src/utils/model_optimizer.py using shap.TreeExplainer — which computes exact Shapley values for tree-based models in polynomial time, avoiding the approximation overhead of KernelSHAP. A 500-sample background dataset (sampled from the training corpus with random_state=42) is used to establish the reference distribution for Shapley marginalisation:

Python — model_optimizer.py (lines 77–81) 
# background: 500 samples from Top-20 training features
background = shap.utils.sample(
    df[top_features.index],
    args.background_samples,   # default: 500
    random_state=42
)
explainer = shap.TreeExplainer(model, background)
joblib.dump(explainer, args.shap_path)   # → models/shap_explainer.pkl
B. Runtime Explanation Generation

At inference time, the XAI engine (xai_engine.py) loads the pre-built SHAP explainer from disk (warm-up at module import to avoid the per-prediction overhead) and computes feature contributions for each classified flow. Only contributions from the curated Top-20 feature set are reported; the Top-20 list is loaded from models/top_20_features.json, enabling the feature set to be updated without modifying the XAI engine source. The engine returns the top-N contributing features (default N=3) ranked by absolute SHAP value:

Python — xai_engine.py (lines 44–91)
def explain_attack(
    input_vector, feature_names,
    attack_class_index=1,   # index of attack class to explain
    top_n=3               # top contributing features to return
) -> List[dict]:
    shap_values = _EXPLAINER(vector)
    attack_values = shap_values[attack_class_index][0]

    contributions = []
    for name, value in zip(feature_names, attack_values):
        if name not in top_features: continue
        direction = "positive" if value >= 0 else "negative"
        contributions.append({"feature": name, "impact": abs(value)})

    # Sort by |SHAP|, return top-N with magnitude classifications:
    # High ≥ 0.5 · Medium ≥ 0.2 · Low < 0.2
    return contributions.sorted(key=impact)[:top_n]

Example output for a detected Port Scan flow: [{"feature": "Flow Packets/s", "impact": "High (positive |SHAP|=0.714)"}, {"feature": "FIN Flag Count", "impact": "Medium (negative |SHAP|=0.321)"}, {"feature": "Fwd Packet Length Mean", "impact": "Low (positive |SHAP|=0.089)"}]. This structured output is passed to log_attack() and stored in alerts.db, making every alert fully auditable from the dashboard.

✅ SOC Operational Value

SHAP-based attribution transforms the production NIDS from a black-box alert generator into a decision-support system: instead of "ALERT: attack from 192.168.1.105", the analyst sees "ALERT: attack from 192.168.1.105 — primary indicator: Flow Packets/s anomaly (High impact), secondary: abnormal FIN flag count". This aligns with SOC Level-2 analyst workflows and significantly reduces mean time to investigate (MTTI) for confirmed attacks.

7.5

Containerised Infrastructure — Docker Compose

The Apache Kafka message bus (required by the Live Bridge and AI Consumer) is deployed as a two-container Docker Compose stack, isolating the message infrastructure from the host Python environment and enabling one-command deployment: docker-compose up -d.

docker-compose.yml
services:
  zookeeper:
    image: confluentinc/cp-zookeeper:7.4.0
    container_name: network-ips-zookeeper
    environment:
      ZOOKEEPER_CLIENT_PORT: 2181
      ZOOKEEPER_TICK_TIME:  2000
    ports:
      - "2181:2181"

  kafka:
    image: confluentinc/cp-kafka:7.4.0       # Confluent Platform 7.4 (Apache Kafka 3.4)
    container_name: network-ips-kafka
    depends_on: [zookeeper]
    ports:
      - "9092:9092"                          # external listener: 127.0.0.1:9092
    environment:
      KAFKA_BROKER_ID:                    1
      KAFKA_ADVERTISED_LISTENERS:         PLAINTEXT://127.0.0.1:9092,PLAINTEXT_INTERNAL://kafka:29092
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1

The dual-listener setup separates external (host process) and internal (container-to-container) communication paths. PLAINTEXT://127.0.0.1:9092 serves the Python producer (live_bridge.py) and consumer (kafka_consumer.py) running on the host. Pinning the Confluent Platform image to version 7.4.0 (Apache Kafka 3.4 compatible) ensures that the container runtime is reproducible across different deployment hosts.

B. Real-Time Dashboard Observability

For production monitoring, the system includes a web dashboard (src/dashboard/app.py, Streamlit-based) that reads from data/live_captured_traffic.csv and alerts.db in near-real-time, displaying: live detection rate per class, recent alert feed (timestamp, src IP, confidence, model used), throughput (messages/sec), and model hot-swap control. This provides operators with a single-pane-of-glass view of system health without requiring any command-line access.

📊 Continuous Model Quality Monitoring

The consumer's STATS dictionary tracks total_processed, attacks_detected, clean_traffic, and errors in real-time, printing a statistics summary every 50 messages or 30 seconds. A rising attacks_detected / total_processed ratio during a known benign-traffic period indicates a false positive spike — an operational signal that model recalibration or threshold adjustment may be required.

Section 7 Summary — MLOps & Engineering Practices

Reproducibility (§7.1): Seed=42 applied at all stochastic stages; leakage-free scaler serialised at fit time; 13-package requirements.txt; centralised src/config.py → TOP_FEATURES; PyCaret session_id=42.

Artifact registry (§7.2): 15 artefacts in models/ — every model paired with a .json config snapshot; scalers version-matched to model type; active_model.txt controls hot-swap.

Data integrity (§7.3): 8-check automated audit gate (feature count, leakage detection, forbidden identifier, NaN/inf, class distribution, scaling range, dtype, summary). Separate LSTM array audit verifies 3D shape + float32 + leakage-free sequencing.

Explainability (§7.4): SHAP TreeExplainer — 500-sample background from training data; top-N feature attribution by |SHAP| ranked High/Medium/Low; output logged to alerts.db for per-alert SOC audit trail.

Infrastructure (§7.5): Docker Compose — Confluent Kafka 7.4.0 + Zookeeper; pinned image versions; dual-listener for host/container isolation; one-command startup.

Observability (§7.6): Table 17: 9 automated artefact types (confusion matrices, ROC curves, feature importance, training history, DT rule exports, latency benchmarks); Streamlit dashboard for live monitoring; STATS counter tracks live FPR signals.
